# zynq-axi-lite

## Goal

Provide a fully reproducible end-to-end reference showing how to connect Linux userspace (`/dev/mem`) running on the ARM PS to a custom AXI4-Lite peripheral in the PL using only officially supported AMD Vivado flows.

This repository intentionally avoids custom tooling. Only officially supported AMD tools are used:
- Vivado 2026.1 (synthesis, implementation, IP Packager, SmartConnect)
- PetaLinux 2026.1
- Linux `/dev/mem` via `busybox devmem`

No custom AXI bridge is implemented.

## Architecture

```
Linux userspace (busybox devmem /dev/mem)
    │
ARM Cortex-A53 (PS)
    │  M_AXI_HPM0_FPD  (AXI4, 32-bit data, base 0xA0000000)
SmartConnect  (1 master → 1 slave, AXI4→AXI4-Lite conversion internal)
    │  AXI4-Lite
axi_regs256  (packaged IP, AXI4-Lite slave)
    │
    reg[0]     = 0xA0100001  read-only ping constant
    reg[1-255] = read/write loopback  ← address decode under investigation
```

---

## What this template does

A minimal Vivado block design wires the Zynq UltraScale+ PS to a custom AXI4-Lite slave (RTL supports 256 registers; hardware-verified only for reg0).

SmartConnect handles AXI4 → AXI4-Lite conversion internally.  
No `axi_protocol_converter` is used — that IP caused permanent write hangs (see Known Issues).

The PL slave is a **Vivado-packaged IP** (`axi_regs256 v1.0`), not a module reference.  
The packaged IP is generated by `1.GenerateAxiIp.ps1` and stored in `out/ip/`.

## Known issues (hardware-verified)

### reg0 PASS, reg1–reg255 FAIL — returns 0x00000000

**Status: confirmed on hardware. Root cause hypothesis under investigation.**

reg0 (ping constant `0xA0100001`) is correctly returned — the AXI path is alive.  
reg1–reg255 write and read back `0x00000000` instead of the written value.

**Current leading hypothesis:** Vivado's `create_peripheral` generates `C_S_AXI_ADDR_WIDTH=4`
in the IP template, giving only 16 bytes of address decode (4 registers).  
Writes to reg1+ are accepted by the bus (no hang, BVALID fires) but the slave
address decoder may silently truncate the address to bits [3:0].

**Attempted fix:** patch `C_S_AXI_ADDR_WIDTH` to 10 in the generated HDL and
`component.xml` after `write_peripheral`. The patch applies correctly (verified in
log and in synthesized netlist), but hardware behaviour is unchanged.
This suggests Vivado may be re-deriving the address decode from `component.xml`
address space declarations independently of the HDL parameter.

**Simulation result:** XSim simulation (step 6) passes all 7 tests including
reg1, reg255, and boundary cases — confirming the RTL itself is correct.
The discrepancy appears during Vivado IP packaging and/or address mapping
through the PS→PL AXI infrastructure. Root cause not yet confirmed by experiment.

### axi_protocol_converter causes permanent write hangs

Inserting `axi_protocol_converter` (SI=AXI4, MI=AXI4LITE, TRANSLATION_MODE=2)
between SmartConnect and the slave causes all AXI writes to hang the ARM CPU
permanently (Linux RCU stall, board unresponsive). Reads work fine.  
Removing the converter and connecting SmartConnect directly to the packaged IP
fixes the hang. SmartConnect handles AXI4→AXI4-Lite conversion internally.

## Reproduce — step by step

### Prerequisites

- Windows PC with Vivado 2026.1 installed (`C:\AMDDesignTools\2026.1\`)
- ZCU102 board powered on, JTAG USB cable connected to PC
- ZCU102 booted into PetaLinux from SD card, reachable at a static IP
- SSH key installed on board for passwordless access
- PowerShell 7+

### 1. Edit environment variables

Open `0.Setup.ps1` and set your values:

```powershell
$env:VIVADO      = "C:\AMDDesignTools\2026.1\Vivado\bin\vivado.bat"
$env:BOARD_IP    = "192.168.0.93"          # your board IP
$env:JTAG_URL    = "localhost:3121/xilinx_tcf/Digilent/<your-serial>"
$env:BOARD_USER  = "petalinux"
```

Then dot-source it (required before every other script):

```powershell
. .\0.Setup.ps1
```

### 2. Generate the AXI4-Lite slave IP

```powershell
.\1.GenerateAxiIp.ps1
```

Runs Vivado to package `rtl/axi_regs256.v` as IP into `out/ip/`.  
Patches `C_S_AXI_ADDR_WIDTH` from 4 → 10 in the generated files.

### 3. Build the bitstream

```powershell
.\2.BuildBitstream.ps1
```

Synthesises and implements the block design (~20 min).  
Output: `out/axi_test.bit` and `out/axi_test.xsa`.

### 4. Build PetaLinux SD card image

```powershell
.\3.BuildPetaLinux.ps1
```

Runs inside Docker (`petalinux-zcu102:2026.1`).  
Output: `out/petalinux-sdimage.wic` (write to SD with balenaEtcher or Rufus or `dd`).

Write to SD card (Linux):
```bash
dd if=out/petalinux-sdimage.wic of=/dev/sdX bs=4M status=progress
```

Boot the ZCU102 from SD. Connect with PuTTY (or any serial terminal) at 115200 baud to the board's USB UART port. When prompted, enter username `petalinux` — on first boot PetaLinux immediately asks you to create a new password. Use that password for all subsequent SSH logins.

### 5. Program the FPGA via JTAG

```powershell
.\4.ProgramFpga.ps1
```

Programs `out/axi_test.bit` to the PL via Vivado hw_server.  
Board must be booted into Linux first (JTAG does not reset the PS).

### 6. Run hardware tests over SSH

```powershell
.\5.RunDevmemTest.ps1
```

Expected result on current hardware:

```
  PASS  reg0   0xa0000000 [read]           : 0xA0100001
  FAIL  reg1   0xa0000004 [write/read]     : got '0x00000000'  expected '0xDEADBEEF'
  FAIL  byteen 0xa0000008 [byte-write/read]: got '0x00000000'  expected '0xFFFFFFAB'
  FAIL  reg254 0xa00003f8 [write/read]     : got '0x00000000'  expected '0xDEADBEEF'
  FAIL  reg255 0xa00003fc [write/read]     : got '0x00000000'  expected '0xCAFEBABE'
```

### 7. Run RTL simulation (no board needed)

```powershell
.\6.SimTest.ps1
```

Runs XSim locally. All 7 tests pass in simulation, confirming the RTL itself is correct.
The discrepancy appears during Vivado IP packaging and/or address mapping through the PS→PL AXI infrastructure.

---

## Pass / Fail criteria

| Test | Expected | Fail means |
|------|----------|------------|
| Ping constant reg0 | `0xA0100001` | AXI path dead — clock/reset or addressing wrong |
| Write loopback reg1–255 | value written | Address decode truncated (ADDR_WIDTH bug) |
| Byte enable | partial update | WSTRB not reaching slave |
| Boundary reg254/255 | values written | Address decode wraps or truncates |

## Troubleshooting

**`devmem` returns 0 for reg1+** — ADDR_WIDTH=4 bug. See Known Issues above.

**`devmem2` hangs** — BVALID never asserted. Remove `axi_protocol_converter` — see Known Issues.

**`/dev/mem: Operation not permitted`** — Use `sudo busybox devmem` (available on this PetaLinux image).
