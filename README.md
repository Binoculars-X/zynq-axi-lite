# zynq-axi-lite

> **Just want the IP, not the whole build pipeline?** See
> [`ip-export/README.md`](ip-export/README.md) for a standalone,
> hardware-verified `axi_regs256.sv` you can drop into any project without
> running steps 0–5.

## Goal

Provide a fully reproducible end-to-end reference showing how to connect Linux userspace (`/dev/mem`) running on the ARM PS to a custom AXI4-Lite peripheral in the PL using only officially supported AMD Vivado flows.

This repository intentionally avoids custom tooling. Only officially supported AMD tools are used:
- Vivado 2026.1 (synthesis, implementation, IP Packager, SmartConnect)
- Vivado 2022.2 (used only for `1.GenerateAxiIp.ps1`'s `create_peripheral` step — see `docs/known-bug-discussion.md`)
- PetaLinux 2026.1
- Linux `/dev/mem` via `busybox devmem`

No custom AXI bridge is implemented.

## Architecture

```
Linux userspace (busybox devmem /dev/mem)
    │
ARM Cortex-A53 (PS)
    │  M_AXI_HPM0_LPD  (AXI4, 32-bit data, base 0x80000000)
SmartConnect  (1 master → 1 slave, AXI4→AXI4-Lite conversion internal)
    │  AXI4-Lite
axi_regs256  (packaged IP, AXI4-Lite slave)
    │
    reg[0]     = 0xA0100001  read-only ping constant
    reg[1-255] = read/write loopback  (hardware-verified, full range)
```

**Why `M_AXI_HPM0_LPD` and not `M_AXI_HPM0_FPD`:** `M_AXI_HPM0_FPD` is a wide
(128-bit) high-performance port; despite setting
`CONFIG.PSU__MAXIGP0__DATA_WIDTH {32}` (a Zynq-7000-era property name not
honored on UltraScale+ for this port), SmartConnect silently inserted a
128→32 downsize converter that corrupted `WDATA`/`WSTRB` on narrow writes
(confirmed via ILA capture — the wire showed `WDATA=0, WSTRB=0` for a write
that should have carried `0xDEADBEEF`/`0xF`). `M_AXI_HPM0_LPD` is natively
32-bit and avoids this conversion path entirely. Its valid aperture is fixed
at `0x80000000` (512MB), hence the base address change from the original
`0xA0000000` plan.

## What this template does

A minimal Vivado block design wires the Zynq UltraScale+ PS (via
`M_AXI_HPM0_LPD`) to a custom AXI4-Lite slave (RTL supports 256 registers;
hardware-verified for the full register range, not just reg0).

SmartConnect handles AXI4 → AXI4-Lite conversion internally.  
No `axi_protocol_converter` is used — that IP caused permanent write hangs (see Known Issues).

The PL slave is a **Vivado-packaged IP** (`axi_regs256 v1.0`), not a module reference.  
The packaged IP is generated by `1.GenerateAxiIp.ps1` and stored in `out/ip/` (build artifact, not committed — see `.gitignore`).

## Known issues (hardware-verified, now resolved)

### reg1–reg255 write corruption via M_AXI_HPM0_FPD (FIXED)

**Status: root-caused and fixed.** See `research-history.md` for the full
investigation timeline (IP-generation-version hypotheses, address-range
hypotheses, all ruled out) and the ILA capture that proved the actual cause.

`reg0` (ping constant `0xA0100001`) always worked — the AXI path itself was
alive. `reg1`–`reg255` writes appeared to complete the AXI handshake (BVALID
fired, response OKAY) but the value never landed. An ILA on the slave's
`S00_AXI` write channel showed `WDATA=0x00000000, WSTRB=0x0` on the wire for
a write that should have carried `0xDEADBEEF`/`0xF` — i.e. the PS master
itself was driving zeros, not the slave RTL misbehaving.

**Fix:** switched the PS master port from `M_AXI_HPM0_FPD` to
`M_AXI_HPM0_LPD` (see Architecture above). Confirmed fixed — all 256
registers, byte-enable, and boundary registers now read/write correctly.

### axi_protocol_converter causes permanent write hangs

Inserting `axi_protocol_converter` (SI=AXI4, MI=AXI4LITE, TRANSLATION_MODE=2)
between SmartConnect and the slave causes all AXI writes to hang the ARM CPU
permanently (Linux RCU stall, board unresponsive). Reads work fine.  
Removing the converter and connecting SmartConnect directly to the packaged IP
fixes the hang. SmartConnect handles AXI4→AXI4-Lite conversion internally.

## Reproduce — step by step

### Prerequisites

- Windows PC with Vivado 2026.1 installed (`C:\AMDDesignTools\2026.1\`)
- Vivado 2022.2 also installed (used only for IP generation, see `0.Setup.ps1` -> `VIVADO_IP_GEN`)
- ZCU102 board powered on, JTAG USB cable connected to PC
- ZCU102 booted into PetaLinux from SD card, reachable at a static IP
- SSH key installed on board for passwordless access
- PowerShell 7+

### 1. Edit environment variables

Open `0.Setup.ps1` and set your values:

```powershell
$env:VIVADO        = "C:\AMDDesignTools\2026.1\Vivado\bin\vivado.bat"
$env:VIVADO_IP_GEN = "C:\Xilinx\Vivado\2022.2\bin\vivado.bat"  # only used by 1.GenerateAxiIp.ps1
$env:BOARD_IP      = "192.168.0.93"          # your board IP
$env:JTAG_URL      = "localhost:3121/xilinx_tcf/Digilent/<your-serial>"
$env:BOARD_USER    = "petalinux"
```

Then dot-source it (required before every other script):

```powershell
. .\0.Setup.ps1
```

### 2. Generate the AXI4-Lite slave IP

```powershell
.\1.GenerateAxiIp.ps1
```

Runs Vivado to package `rtl/axi_regs256.v` as IP into `out/ip/`.  
Patches `C_S_AXI_ADDR_WIDTH` from 4 → 10 in the generated files.

### 3. Build the bitstream

```powershell
.\2.BuildBitstream.ps1
```

Synthesises and implements the block design (~20 min).  
Output: `out/axi_test.bit` and `out/axi_test.xsa`.

### 4. Build PetaLinux SD card image

```powershell
.\3.BuildPetaLinux.ps1
```

Runs inside Docker (`petalinux-zcu102:2026.1`).  
Output: `out/petalinux-sdimage.wic` (write to SD with balenaEtcher or Rufus or `dd`).

**⚠️ Requires two large files that are gitignored and must be downloaded
separately (see `petalinux-build/README.md`):**

| File | Size | Where to get it |
|------|------|-----------------|
| `petalinux-build/petalinux-v2026.1-06061130-installer.run` | ~8 GB | Find on the AMD/Xilinx downloads site (PetaLinux Tools 2026.1) |
| `petalinux-build/xilinx-zcu102-v2026.1-06092129.bsp` | ~2 GB | Same site, Board Support Packages → ZCU102 |

An AMD/Xilinx account is required (free). Place both files directly in
`petalinux-build/` before running `docker build`/`3.BuildPetaLinux.ps1`.

Write to SD card (Linux):
```bash
dd if=out/petalinux-sdimage.wic of=/dev/sdX bs=4M status=progress
```

Boot the ZCU102 from SD. Connect with PuTTY (or any serial terminal) at 115200 baud to the board's USB UART port. When prompted, enter username `petalinux` — on first boot PetaLinux immediately asks you to create a new password. Use that password for all subsequent SSH logins.

### 5. Program the FPGA via JTAG

```powershell
.\4.ProgramFpga.ps1
```

Programs `out/axi_test.bit` to the PL via Vivado hw_server.  
Board must be booted into Linux first (JTAG does not reset the PS).

### 6. Run hardware tests over SSH

```powershell
.\5.RunDevmemTest.ps1
```

Expected result:

```
  PASS  reg0   0x80000000 [read]            : 0xA0100001
  PASS  reg1   0x80000004 [write/read]      : 0xDEADBEEF
  PASS  byteen 0x80000008 [byte-write/read] : 0xFFFFFFAB
  PASS  reg254 0x800003f8 [write/read]      : 0xDEADBEEF
  PASS  reg255 0x800003fc [write/read]      : 0xCAFEBABE
=== ALL TESTS PASSED ===
```

Or use the single-entry-point orchestration script instead of running 1/2/4/5
individually:

```powershell
. .\0.Setup.ps1
.\RunAll.ps1
```

### 7. Run RTL simulation (no board needed)

```powershell
.\6.SimTest.ps1
```

Runs XSim locally. All 7 tests pass in simulation, confirming the RTL itself
is correct (this was true even before the `M_AXI_HPM0_LPD` fix — the bug was
never in the RTL, see `research-history.md`).

## Pass / Fail criteria

| Test | Expected | Fail means |
|------|----------|------------|
| Ping constant reg0 | `0xA0100001` | AXI path dead — clock/reset or addressing wrong |
| Write loopback reg1–255 | value written | Master/interconnect data-path issue (see Known Issues — this was the `M_AXI_HPM0_FPD` width-conversion bug) |
| Byte enable | partial update | WSTRB not reaching slave |
| Boundary reg254/255 | values written | Address decode wraps or truncates |

## Troubleshooting

**`devmem` returns 0 for reg1+** — make sure the block design uses `M_AXI_HPM0_LPD`, not `M_AXI_HPM0_FPD`. See Known Issues and `research-history.md`.

**`devmem2` hangs** — BVALID never asserted. Remove `axi_protocol_converter` — see Known Issues.

**`/dev/mem: Operation not permitted`** — Use `sudo busybox devmem` (available on this PetaLinux image).
